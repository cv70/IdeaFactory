# Idea Factory 产品设计文档

> 版本：v1
> 日期：2026-03-13
> 输入依据：`README.md` + 设计评审结论

## 1. 产品定义

`Idea Factory` 应被重新定义为一个 `Graph-First, Research-Augmented Exploration OS`。

它不是点子生成器，不是行业模板库，也不是聊天式研究助手。它的核心职责是把一个模糊主题持续推进为一张可生长、可追溯、可物化的探索图，并从图中产出高价值的 idea 资产。

核心原则：

- 统一：不同领域共享同一套探索抽象，不按科研、创业、产品、内容拆流程
- 深度优先：第一优先级不是快速出结果，而是把主题推进到更深的问题、假设和机会结构
- 图优先：所有输入先进入图，所有输出从图中投影，不直接绕过图生成答案
- 研究增强：纯推理、上下文注入、外部研究三种深度来源都应被支持
- 双层交付：既有通用探索引擎，也有一个标准 `reference workbench`

## 2. 问题陈述

现有 idea 工具普遍有四个问题：

- 结果是一次性文本，无法继续演化
- 缺少统一中间模型，不同领域依赖不同模板
- 生成结果很快，但探索结构浅，用户很难知道下一步该往哪里深挖
- 外部资料、用户上下文与生成结果之间缺少稳定映射，无法形成长期资产

`Idea Factory` 的目标不是替代搜索、聊天或写作工具，而是补上“探索结构化”这一层，把主题从输入变成一个可持续增长的认知资产。

## 3. 产品定位

产品定位为：

- 一个面向多类创新任务的通用探索操作系统
- 一个高自治的探索引擎，而非强交互问答工具
- 一个开源内核 + 云端托管的双轨产品
- 一个既能服务终端使用者，也能服务开发者二次集成的能力底座

第一核心用户不绑定单一行业角色，而是面向需要系统性发现方向、收敛路径、沉淀思考结构的各类知识工作者与产品构建者。

## 4. 北极星与成功标准

第一版北极星不是“生成多少条 idea”，而是“能否持续产出值得继续推进的探索路径”。

建议成功标准：

- 系统能把主题推进到用户可感知的深层结构，而非停留在表层 brainstorming
- 多轮运行后，图结构能稳定收敛，而不是不断堆叠重复节点
- 不同来源的上下文最终能在同一张图中对齐
- 物化产物能追溯到问题、假设、证据和路径
- 用户更常做出的动作不是“重新来一批”，而是“继续推进这条路径”

建议核心指标：

- 平均有效探索深度
- 高价值路径被继续展开的占比
- 从主题输入到出现“值得继续”的路径所需轮次
- 每轮新增结构与重复结构的比例
- artifact 被二次使用、继续追问或导出的比例

## 5. 产品边界

### 5.1 做什么

- 把模糊主题转化为结构化探索图
- 用高自治 runtime 自动扩展问题、张力、假设、机会与 idea
- 接入外部研究和用户上下文作为探索燃料
- 对探索路径进行评估、排序、压缩和物化
- 通过标准工作台展示图的不同投影

### 5.2 不做什么

- 不做按领域切分的主流程产品
- 不做聊天框驱动的主体验
- 不做重型可视化图编辑器
- 不做第一版就自动生成长篇商业计划、PRD、论文初稿
- 不做重型协作与权限体系
- 不做模板市场或 prompt 工坊

## 6. 核心产品模型

产品层应建立在三层统一模型上：

- `Exploration Graph`：系统唯一真相层
- `Exploration Runtime`：驱动图增长、评估、反思和物化的自治运行层
- `Research Substrate`：外部资料、历史 session、用户文档、检索结果等上下文燃料层

系统关系为：

`Context / Research Inputs -> Runtime -> Graph -> Materializations`

这里最关键的产品决策是：

- 系统产物不是一次回答，而是一张不断演化的探索图
- 系统交付给用户的不是“最终答案”，而是“当前最值得继续的路径与其依据”

## 7. 参考产品形态

`reference workbench` 不应只是一个前端原型，而是引擎能力的标准展示层。

它的职责是：

- 验证引擎是否真的具备深度探索能力
- 给开发者和使用者提供一套理解系统的标准交互方式
- 展示同一张图的多种投影，而非暴露底层数据库结构

建议 workbench 验证四类核心视图：

- `Map View`：方向、机会簇、未知区和结构空白
- `Path View`：某条从问题到 idea 的推演链
- `Run View`：某次运行做了什么、为什么这样做
- `Artifact View`：已物化产物及其来源追溯

## 8. 核心交互流

第一版主交互流建议重写为四段：

### 8.1 Seed

用户输入探索种子：

- 主题
- 目标
- 约束
- 可选上下文源
- 期望输出类型
- 探索预算

### 8.2 Run

系统以高自治方式启动一轮 exploration run：

- 解释主题
- 构建初始问题空间
- 判断研究缺口
- 注入外部研究或用户上下文
- 形成高价值路径候选

### 8.3 Review

系统先交付一个“探索状态”而不是答案页：

- 当前高价值路径
- 低置信或高争议路径
- 关键未知点
- 证据缺口
- 推荐下一步动作

### 8.4 Materialize

系统或用户选择路径，将其物化为特定产物，例如：

- 机会地图
- 研究方向包
- 方案空间摘要
- 选题簇
- idea 集合
- 下一步实验建议

## 9. MVP 设计

第一版 MVP 要验证的是能力底座，而不是完整产品壳。

### 9.1 引擎必需能力

- exploration session 创建与运行
- graph 增量构建与版本化
- context source 注入
- 多阶段 run orchestration
- path evaluation 与 ranking
- artifact materialization
- feedback 写回图
- run 可追溯记录

### 9.2 产品演示必需能力

- session 创建页
- run 控制台
- `map/path/run/artifact` 四视图
- 节点路径展开与重跑
- 基础反馈动作
- artifact 导出与引用追溯

## 10. 核心差异化

相较于传统点子工具、模板工具和聊天式助手，本产品的差异化在于：

- 有统一探索图，而不是一次性结果
- 有自治 runtime，而不是 prompt 拼接
- 有研究增强输入，而不是只依赖单轮推理
- 有路径级评估与推荐，而不是只生成更多卡片
- 有 artifact 物化层，而不是只输出文本

## 11. 风险与约束

第一版主要风险：

- 图模型过重，导致使用成本高于用户收益
- runtime 深度不足，最终退化为普通 brainstorm
- research 增强处理不统一，导致系统变成上下文拼接器
- workbench 过度追求界面完整性，反而掩盖引擎能力不足

应对原则：

- 所有能力围绕“探索深度”服务
- 以图为唯一真相层，避免旁路逻辑
- 先验证结构深度，再扩展行业化包装
- 坚持双层架构，防止前台原型绑架内核

## 12. 产品路线建议

建议分三阶段推进：

### 阶段一：引擎成型

目标是建立稳定的图模型、run 模型、research 注入模型和 artifact 模型。

### 阶段二：参考工作台验证

目标是验证用户是否能理解系统为何推进某条路径，以及是否愿意继续沿路径深挖。

### 阶段三：平台化扩展

目标是开放 API、SDK、strategy 配置、research adapter 和行业化 artifact 模板。

## 13. 一句话总结

`Idea Factory` 应从“探索工作台前端原型”升级为一个 `Graph-First, Research-Augmented Exploration OS`：它以统一探索图为核心真相层，以高自治 runtime 驱动问题空间与机会结构的持续增长，再将高价值路径物化为可追溯、可扩展的 idea 资产。
