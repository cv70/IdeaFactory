## 1. 文档职责与边界

本文件只回答三件事：

- 用户为什么要用 Idea Factory
- 用户如何围绕 `workspace` 与 `方向地图` 完成探索治理
- 产品成功如何被衡量

本文件不定义：

- `run / plan / task / balance` 等内部执行机制
- 子系统部署、存储一致性、故障恢复策略
- API 字段与错误码

对应内容见：

- 技术设计：[idea-factory-technical-design.md](./idea-factory-technical-design.md)
- 系统架构：[idea-factory-system-architecture.md](./idea-factory-system-architecture.md)
- OpenAPI 契约：[idea-factory-openapi.yaml](./idea-factory-openapi.yaml)

## 2. 产品目标

Idea Factory 的产品目标是：让用户在长期存在的 `workspace` 中，把模糊主题持续推进为可比较、可追溯、可继续深挖的 `方向地图`。

产品不是：

- 聊天式答案生成器
- 一次性脑暴列表工具
- 通用图编辑器

## 3. 用户角色与职责

### 3.1 用户角色

用户是 `workspace` 的委托者与治理者，核心职责：

- 提供主题、目标、约束、预算
- 在运行中提交方向性 `intervention`
- 对地图分支做保留、压制、继续深挖判断

### 3.2 系统角色

系统默认主驾驶，核心职责：

- 持续推进探索
- 维护方向结构的可解释性
- 在用户干预后调整后续探索重点

## 4. 核心闭环

唯一主闭环：

`模糊主题 -> 自治探索 -> 方向地图 -> 用户干预治理 -> 新一轮探索`

对用户可见的关键结果：

- 当前有哪些方向
- 方向之间如何比较
- 最近哪些变化最重要
- 干预后系统是否已吸收并改变重心

## 5. 主界面信息架构（Map-First）

进入 workspace 后默认界面包含四层：

1. 地图主画布：方向结构、热点分支、折叠弱方向
2. 轻量状态层：当前推进焦点与最近变化摘要
3. 分支详情层：该方向的依据、判断与未解决问题
4. 干预入口层：提交高层方向指令

`run` 与轨迹信息是次级解释视图，不作为首页主入口。

## 6. Intervention 产品契约（用户视角）

用户提交 intervention 后，产品必须返回三类反馈：

- 已吸收：系统是否接受并进入处理
- 已变化：地图结构出现了哪些关键变化
- 已转向：后续探索重心如何调整

如果这三类反馈缺失，用户会把系统视为黑盒。

## 7. 北极星与能力验收（产品层）

北极星：用户是否持续沿同一 workspace 深挖高价值方向，而不是反复重开主题。

产品层能力清单验收：

- 能看到至少 2-3 条可比较方向，而非一堆平铺想法
- 能解释方向变化，不只展示新内容
- 干预后能观察到可见结构变化
- 用户可基于方向继续物化为 artifact

## 8. 术语主入口与引用规则

产品默认术语：`workspace`、`方向地图`、`intervention`、`artifact`。

以下术语仅作解释引用，不在本文件定义：`run`、`execution_plan`、`agent_task`、`balance_state`、`trace_summary`。

这些术语的定义唯一来源为技术设计与 OpenAPI。

## 9. 一句话总结

Idea Factory 的产品面应始终围绕“用户治理方向地图”，而不是“用户操作运行内核”。
